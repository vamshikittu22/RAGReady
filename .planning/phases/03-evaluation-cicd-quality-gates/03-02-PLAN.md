  <action>
    Create `.github/workflows/ci.yml` with the following job structure:

    **Trigger:** `on: [push, pull_request]` targeting `main` branch.

    **Job 1: lint-and-typecheck**
    - Runs on: `ubuntu-latest`
    - Python: `3.12` (not 3.14 — GitHub Actions may not have 3.14 yet; 3.12 is the minimum supported version per pyproject.toml)
    - Steps:
      1. Checkout code
      2. Set up Python 3.12
      3. Install dependencies: `pip install -e ".[dev]"`
      4. Run ruff: `ruff check src/ tests/`
      5. Run mypy: `mypy src/ --ignore-missing-imports` (use --ignore-missing-imports since some deps may lack type stubs)

    **Job 2: unit-tests**
    - Runs on: `ubuntu-latest`
    - Python: `3.12`
    - Steps:
      1. Checkout code
      2. Set up Python 3.12
      3. Install dependencies: `pip install -e ".[dev]"`
      4. Run unit tests: `pytest tests/unit/ -v --tb=short`

    **Job 3: integration-tests**
    - Runs on: `ubuntu-latest`
    - Python: `3.12`
    - Needs: `unit-tests` (run after unit tests pass)
    - Steps:
      1. Checkout code
      2. Set up Python 3.12
      3. Install dependencies: `pip install -e ".[dev]"`
      4. Cache HuggingFace models: Use `actions/cache@v4` with path `~/.cache/huggingface` and key based on `hashFiles('pyproject.toml')`
      5. Run integration tests: `pytest tests/integration/ -v --tb=short`

    **Job 4: evaluation**
    - Runs on: `ubuntu-latest`
    - Python: `3.12`
    - Needs: `integration-tests` (run after integration tests pass)
    - Steps:
      1. Checkout code
      2. Set up Python 3.12
      3. Install dependencies: `pip install -e ".[dev,eval]"`
      4. Cache HuggingFace models: Use `actions/cache@v4` with path `~/.cache/huggingface` and key `huggingface-${{ hashFiles('pyproject.toml') }}`
      5. Create reports directory: `mkdir -p reports`
      6. Run evaluation tests: `pytest tests/evaluation/ -v --tb=short -m "evaluation and not slow and not ollama"` — this runs only deterministic + HHEM metrics, skipping LLM-dependent tests
      7. Upload evaluation report artifact:
         ```yaml
         - uses: actions/upload-artifact@v4
           if: always()
           with:
             name: evaluation-report
             path: reports/
             retention-days: 30
         ```

    **Important CI notes:**
    - Each job should use `pip install -e ".[dev]"` (editable install with dev deps)
    - The evaluation job also needs `.[eval]` for ragas dependency
    - Evaluation tests that need Ollama are marked with `@pytest.mark.ollama` and excluded from CI
    - Quality gate tests ARE run in CI — they assert thresholds and block the PR on failure
    - The `if: always()` on artifact upload ensures reports are saved even if tests fail

    **Update .gitignore:**
    Add `reports/` directory if not already present. Also ensure `data/` and `*.pkl` are gitignored (they should already be from Phase 1).

  <action>
    Create `scripts/benchmark.py` — a standalone CLI script for running the naive vs hybrid retrieval benchmark comparison.

    **Purpose:** Generates a side-by-side comparison report proving hybrid retrieval outperforms naive dense-only retrieval. This script can be run locally for detailed analysis.

    **Implementation:**

    ```python
    """Benchmark: Naive (dense-only) vs Hybrid retrieval comparison.

    Runs the golden dataset through both DenseRetriever and HybridRetriever,
    computes metrics for each, and generates a comparison report.

    Usage: python scripts/benchmark.py [--output reports/]
    """
    ```

    Steps:
    1. Parse CLI args (--output directory, defaults to `reports/`)
    2. Load Settings with tmp_path-style temp directory for indexes
    3. Create IngestionPipeline and ingest all 3 test fixture documents
    4. Create both retrievers:
       - DenseRetriever (naive baseline) from pipeline.chroma
       - HybridRetriever (dense + sparse + RRF) from pipeline.chroma + pipeline.bm25
    5. Load golden dataset from `tests/evaluation/golden_dataset.json`
    6. Run all non-refusal questions through both retrievers
    7. Compute Context Recall and Context Precision for both
    8. Generate comparison report:
       ```json
       {
         "timestamp": "...",
         "metrics": {
           "naive": {"context_recall": 0.65, "context_precision": 0.55},
           "hybrid": {"context_recall": 0.82, "context_precision": 0.75}
         },
         "improvement": {
           "context_recall": "+26%",
           "context_precision": "+36%"
         },
         "winner": "hybrid"
       }
       ```
    9. Print results table to stdout:
       ```
       ┌─────────────────────┬────────┬────────┐
       │ Metric              │ Naive  │ Hybrid │
       ├─────────────────────┼────────┼────────┤
       │ Context Recall      │ 0.65   │ 0.82   │
       │ Context Precision   │ 0.55   │ 0.75   │
       └─────────────────────┴────────┴────────┘
       ```
    10. Save JSON report to output directory

    Use `tempfile.mkdtemp()` for temporary index storage. Clean up after.
    Use argparse for CLI arguments.
    Import metric functions from `tests.evaluation.metrics`.

    **IMPORTANT:** The benchmark script must work standalone (not require pytest). It directly uses the pipeline and retriever classes.
  </action>

    - Script has argparse with --output argument

    - All existing tests still pass: `pytest tests/unit/ tests/integration/ -q`
  </verify>
